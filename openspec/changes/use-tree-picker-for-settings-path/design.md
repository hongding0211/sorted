## Context

当前主界面已经有一套可展开、可折叠的 source tree 浏览模型，但设置页中的 `Destination Root` 仍然通过逐字符编辑路径字符串来完成。两套交互模型并存，一方面增加了学习成本，另一方面也让选择已有目录、逐层确认父子目录关系、避免手输错误变得更困难。

现有实现中，设置页直接把输入写回 `self.settings.destination_root`，随后复用 `validate_settings` 和 `ConfigStore::save` 完成校验与持久化。这意味着如果要改成树状浏览，设计重点不在配置格式，而在于如何为设置页引入一个和 source browser 一致、但不污染主界面状态的目录树浏览上下文。

## Goals / Non-Goals

**Goals:**
- 让设置页的目标路径选择方式与主界面的 source tree 交互一致，包括上下移动、左右展开折叠、确认当前目录。
- 保留现有 destination root 的保存、可创建校验、可写校验和成功提示，不改变配置文件格式。
- 明确区分“当前已保存的 destination root”和“设置页正在浏览/候选的目录”，避免未确认浏览结果直接覆盖设置值。
- 让设置页在帮助文案、焦点高亮和状态反馈上清楚说明当前按键会作用于哪一块内容。

**Non-Goals:**
- 不修改主界面的 source 选择规则，也不改变导入流程中的 source tree 数据来源。
- 不引入新的配置字段、外部依赖或多选目录能力。
- 不改变 date format 的编辑方式，除非为了保持布局一致只做最小程度的展示调整。

## Decisions

### Reuse the existing tree interaction model, but keep separate settings-specific tree state
设置页将复用主界面已经建立的树状浏览交互约定，例如展开/折叠、目录列表扁平化展示和当前选中项高亮；但对应的数据状态需要独立维护，避免 `expanded_sources`、`source_entries`、`source_index` 直接和设置页共用，导致主界面 source 选择被设置页浏览过程污染。

备选方案是把设置页也直接绑定到现有 source tree 状态。这个方案实现更快，但会把“导入来源选择”和“配置目标路径选择”耦合到同一批目录展开状态里，用户从设置页返回主界面后可能看到 source tree 被意外改动，因此不采用。

### Treat directory browsing as a staged selection before save
设置页中的树状浏览应先产生一个候选目录，只有在用户明确确认该目录用于 `Destination Root` 后，才更新设置表单中的值；而真正写入配置文件仍然发生在保存动作。这样可以把“浏览目录”“采用该目录”“持久化设置”分成三个清晰阶段。

备选方案是每次光标移动到目录时就立即覆盖 `destination_root`。这个方案反馈更即时，但会让验证和预览不断抖动，也会使用户难以判断自己是在浏览还是已经修改了设置，因此不采用。

### Keep destination validation in the existing config layer
目录树只负责帮助用户选路径，不重新定义合法性规则。最终保存时仍然通过现有 `validate_destination_root` / `validate_settings` 来判断目录是否可创建、是否可写、是否是文件路径，从而保持主界面路径预览和配置落盘逻辑的一致性。

备选方案是在 UI 层单独实现一套目录合法性判断。这样会造成规则重复，并增加 UI 与核心配置逻辑偏离的风险，因此不采用。

### Surface saved value, candidate value, and help text together
设置页需要同时展示当前已设置的 `Destination Root`、树状目录浏览区域，以及明确的帮助文案，说明 `Enter` 是确认目录还是保存整个设置。这样可以减少“当前选中目录是否已经生效”的歧义。

备选方案是只保留树状列表，不单独展示当前配置值。这个方案会让用户难以判断保存前后具体使用的路径，也不利于验证失败时回退和解释状态，因此不采用。

## Risks / Trade-offs

- [设置页树状态与主界面树状态过于相似，容易在实现时共用错字段] -> 将设置页浏览状态建模为独立字段或独立结构体，并补充从设置页返回主界面后的状态隔离测试。
- [用户可能分不清确认目录和保存设置是两个动作] -> 在设置页帮助文案和状态消息中分别描述“确认当前目录”和“保存设置”的结果。
- [目标路径可能非常深，树状浏览增加更多按键步骤] -> 默认从当前已配置路径附近开始浏览，并保留清晰的当前路径显示，减少重复展开成本。
- [某些目录不可读或加载较慢] -> 沿用主界面目录加载中的状态反馈，并在不可进入时给出明确提示而不是静默失败。

## Migration Plan

这是一次纯交互层变更，不涉及配置文件 schema 迁移。发布时只需确保已有配置仍可加载，并在设置页初始化时把已保存的 destination root 映射成树状浏览的起点或当前值即可；如果新浏览器初始化失败，回退策略应是继续保留现有已保存设置并阻止保存非法值，而不是覆盖配置。

## Open Questions

- 设置页目录树是否需要从文件系统根节点开始浏览，还是优先定位到当前已保存 destination root 的最近可访问父目录。
- `Enter` 在设置页中是“先确认目录，再在 date format 字段保存整个设置”，还是需要为目录确认引入单独快捷键；实现时需要结合现有焦点模型决定最小惊讶方案。
